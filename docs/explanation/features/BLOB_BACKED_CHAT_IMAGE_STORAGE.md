# Blob-Backed Chat Image Storage

## Overview

Enhanced-citation chat images are stored as blob-backed chat artifacts instead of embedding image bytes directly in Cosmos DB message content. Chat rendering continues to use authenticated image routes, so streaming and non-streaming responses receive a lightweight `/api/image/<message_id>` URL.

## Dependencies

- Azure Blob Storage chat containers: `personal-chat` and existing group chat container configuration.
- Enhanced citations setting: `enable_enhanced_citations`.
- Existing authenticated image endpoints for personal and collaborative conversations.

## Technical Specifications

### Architecture

- Uploaded and generated images use blob-backed message metadata when enhanced citations are enabled.
- Message records keep `file_content_source`, `blob_container`, `blob_path`, `mime_type`, and a lightweight authenticated route in `content`.
- Legacy inline and chunked Cosmos image storage remains available when enhanced citations are disabled or for older messages.
- Blob-backed image files are compatible with existing blob-backed message deletion workflows.

### API Endpoints

- `GET /api/image/<message_id>` streams personal or collaboration-visible blob-backed source images after conversation authorization.
- `GET /api/collaboration/conversations/<conversation_id>/images/<message_id>` streams blob-backed source images for authorized collaboration viewers.

## Usage Instructions

No new user configuration is required beyond enabling enhanced citations. Images uploaded to chat or generated by image-capable models render through the same chat UI image flow, but the stored message content remains small.

## Testing and Validation

Functional coverage: `functional_tests/test_chunked_image_storage.py` validates blob-backed image hydration alongside existing chunked image behavior.

Performance consideration: image bytes stream from Blob Storage instead of expanding Cosmos message payloads, reducing message item size and avoiding the Cosmos item-size pressure that large base64 images create.

Known limitation: existing legacy image messages remain in their original inline or chunked shape until they are naturally removed by existing retention or deletion workflows.
