Governance Configuration Guide
Governance controls who can use capabilities and delegated resources. Feature policies decide whether a user can use a class of capability, while delegated item policies decide whether a user can use a specific global endpoint, global agent, or global action.
Governance Boundary: SimpleChat application-level features are governed by identity and app roles. Capabilities and resources configured inside SimpleChat, including agents, actions, endpoints, and delegated items added by admins or users, are governed by in-app governance rules. Use app roles for platform access boundaries, and use in-app governance for resources and capabilities created inside SimpleChat.
Two Layers of Governance
1. Feature Policies

Feature policies answer: Can this user use this type of capability?

  • Personal agents, actions, and endpoints
  • Group agents, actions, and endpoints
  • Global agent, action, and endpoint usage
2. Delegated Item Policies

Delegated item policies answer: Can this user use this exact delegated resource?

  • A specific global endpoint
  • A specific global agent
  • A specific global action
Rule of thumb: The user must pass the feature policy first. Delegated item policies are additive whitelists inside that feature gate; they do not override feature-level governance.
How Feature Policies Work

Feature toggles enable enforcement for a capability. The matching feature policy then decides who passes that check.

Setting What It Governs
Govern Personal Endpoints Whether a user can use model endpoints configured in their personal workspace.
Govern Personal Agents Whether a user can create, view, or use agents in their personal workspace.
Govern Personal Actions Whether a user can create, view, or use actions in their personal workspace.
Govern Group Endpoints Whether a user can use model endpoints configured in group workspaces.
Govern Group Agents Whether a user can use agent capabilities in group workspaces where group agent features are enabled.
Govern Group Actions Whether a user can use action capabilities in group workspaces where group action features are enabled.
Govern Global Endpoints Whether a user can use global model endpoints shared by admins.
Govern Global Agents Whether a user can select or use global agents shared by admins.
Govern Global Actions Whether a user can use global actions shared by admins.
Important: If Allow All is off, add at least one allowed user or allowed group when you intend to restrict access. Empty explicit lists are treated permissively for backward compatibility.
Using Workspaces as Governance Cohorts
A group workspace can be used as a reusable group of users for governance. In this context, the workspace membership is used as an access cohort, not as a content-sharing permission.

This is useful when admins want to pilot personal features with a defined set of people without managing every user ID in every governance policy.

What It Does
  • Uses group workspace membership as a reusable allow-list cohort.
  • Allows members of that workspace to pass selected feature policies.
  • Works for personal feature policies such as personal agents, personal actions, and personal endpoints.
What It Does Not Do
  • It does not grant access to that group workspace's documents.
  • It does not grant access to group agents or group actions by itself.
  • It does not move personal agents into the group workspace.
Example: Personal Agents Pilot
  1. Create a group workspace named Personal Agents Pilot Users.
  2. Add the users who should test personal agents.
  3. Open Governance > Feature Policies.
  4. Edit Personal Agents, turn off Allow All, and add the Personal Agents Pilot Users workspace ID as an allowed group.
  5. Members can now use personal agents in their own personal workspaces. They do not automatically receive access to group documents, group agents, or delegated global resources.
Delegated Item Policies

Delegated item policies are resource-specific checks for global resources. They are evaluated after the matching feature policy passes.

Important: Item delegation cannot grant access by itself. If a user is allowed by an item policy for Agent X but does not pass Govern Global Agents, the user is blocked from Agent X.
Global Endpoints

Restrict a specific global model endpoint to selected users or groups.

Global Agents

Restrict a specific global agent to selected users or groups.

Global Actions

Restrict a specific global action to selected users or groups.

Tip: Use delegated item policies when a feature is broadly available, but a particular global resource should only be available to a narrower audience.
Evaluation Flow
  1. The backend identifies the signed-in user.
  2. The relevant governance setting is checked. If governance is off for that feature, access is allowed.
  3. The user's governance groups are resolved from group workspace and public workspace memberships.
  4. The feature policy is evaluated against Allow All, allowed users, and allowed groups.
  5. If the request targets a delegated global endpoint, global agent, or global action, matching item policies are evaluated only after the feature policy passes.
  6. If the feature policy blocks the user, item-level delegation is not considered and cannot override the denial.
  7. The backend allows or denies the operation. Frontend filtering only improves the experience and is not the security boundary.
Performance note: Governance decisions use request-level memoization, short-lived process caching, and shared cache-version invalidation when Redis is available.
Common Questions
