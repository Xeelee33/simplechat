Identity setup by provider
  • Azure OpenAI: assign Reader plus Cognitive Services OpenAI User on the Azure OpenAI resource when using managed identity or service principal model discovery.
  • Foundry (classic): assign Foundry User, or Azure AI User where older role names still appear, on the target Foundry project or backing resource.
  • New Foundry: use the same Foundry project access as classic Foundry, then select the New Foundry provider and the project endpoint in this modal.
  • Provider setup: for Foundry project model endpoints, keep OpenAI API Version at endpoint default v1. Use separate endpoints when Grok, Meta/Llama, DeepSeek, OpenAI-compatible, or other model families need different project settings, auth, or manual deployment rows.
  • API Key: use for inference-only endpoints or APIM paths. Model and Foundry project discovery requires managed identity or service principal RBAC.
For APIM, choose the matching provider with API key auth. If using classic Foundry, use Foundry (classic). If using the application-based runtime, use New Foundry.
For Azure OpenAI, paste the resource endpoint.
Project API versions use the v1 format. Choose Custom only when your Foundry project requires a different supported value.
For Foundry project endpoints, Project API Version controls discovery and usually stays v1. OpenAI API Version controls the normalized /openai/v1 inference client and should stay Endpoint default (v1); the /v1 path does not allow an api-version query. Split model families into separate endpoints when they need different project settings or auth. Claude deployments are detected from the model name and use the Anthropic messages protocol.
If the project endpoint includes /api/projects/<project>, the project name is detected automatically.
API key authentication is for inference only. Use a managed identity or service principal for model discovery, or use Add Model and enter the deployment name manually exactly as it appears in Foundry.

Provider selection
Provider Use for Endpoint value
Azure OpenAI Direct Azure OpenAI resource endpoints and Azure OpenAI-compatible APIM paths. Use the Azure OpenAI resource endpoint, such as https://resource.openai.azure.com/.
Foundry (classic) Classic Foundry project model deployments and classic Foundry agent import. Use the Foundry project endpoint. If the URL does not include /api/projects/project-name, fill Foundry Project Name.
New Foundry Application-based Foundry runtime, New Foundry agents, and project model deployments. Use the New Foundry project endpoint. Set Project API Version for discovery, usually v1, and keep OpenAI API Version at endpoint default v1 for the normalized /openai/v1 inference path.
Roles to assign
Scenario Assign to Role and scope
Azure OpenAI with Fetch Models The App Service managed identity, user-assigned managed identity, or service principal enterprise application. Reader plus Cognitive Services OpenAI User on the Azure OpenAI resource.
Azure OpenAI with manually entered model rows The same runtime identity or service principal. Cognitive Services OpenAI User on the Azure OpenAI resource.
Foundry (classic) and New Foundry discovery or invocation The same runtime identity or service principal. Foundry User on the Foundry project or backing resource. In Azure Government or clouds still showing older names, use Azure AI User.
Configuration flow
  1. Assign the role on the correct Azure OpenAI or Foundry resource first.
  2. Select the matching provider in the endpoint modal.
  3. Choose Managed Identity or Service Principal, then fill the visible identity fields.
  4. For Azure OpenAI discovery, enter subscription ID and resource group.
  5. For Foundry providers, enter the project endpoint and project name when the endpoint does not include it.
  6. Keep Foundry Project API Version separate from OpenAI API Version. Create separate endpoints when model families require different inference API versions.
  7. Use Fetch Models and Test Model before saving the endpoint.
